Browser feasibility notes

This public edition is a local-pair prototype. Controller and Player communicate only between browsing contexts on this origin and in the same browser profile. It is not internet remote control and it does not upload audio to a server.

Architecture

Controller → BroadcastChannel → Player. The Controller stores a mutable draft and publishes an immutable revision. The Player enables Web Audio after a local click, prepares the exact revision, reports presence, and acknowledges scheduled commands.

Readiness

Live is available only when the Player link is fresh, its audio context is running, the published revision is prepared, and the Player is visible. Losing readiness returns the Controller to Standby.

Browser limits

  • Every cold Player load needs a local Enable Player click because browsers gate audio autoplay.
  • Fullscreen and screen wake lock are requests, not guarantees, and may be denied or released.
  • The public build accepts files up to 25 MB and 45 seconds, preflights duration, and limits decoded audio memory.
  • Extension names are hints. Only a successful browser metadata read and Web Audio decode accepts a cue.
  • Closing, hiding, suspending, or muting a page or computer can make playback unavailable.
